Research + Wireframes. Lots of Wireframes.

When we started planning, a a clinical trial's process involved fax machines, snail-mailing flash drives, emailing large PDFs, and phone calls. The main goal was to bring several aspects of the clinical trial process online, and to design an experience that our clients wouldn’t be able to live without.

A barebones version of the web app already existed, so we had users and some web analytics. We conducted an exhaustive amount of interviews, combed through the analytics, and proposed a number of new approaches for the interface and functionality. This resulted in numerous sets of site maps and wireframes, which we frequently presented and debated both inside and outside the team.

Design, Iterate, Iterate, Iterate

Once we got a few wireframes nailed down, I began creating higher fidelity versions in Photoshop, and soon thereafter in HTML & CSS. While the app has just a handful of screens, each one has a seemingly infinite number of states and edge cases to account for. Things like oversized photos, missing text fields, and uploads with long file names would often break layouts and force us to rethink and adapt the design. We had the benefit of designing with real content, so used it to test each screen in a myriad of browsers until we were confident in the design's sturdiness.

Fine Tuning & Bulletproofing

With the interface in place and each function working, we focused on improving each individual task. We looked at the the product's four or five user roles (personas, if you will) and revisited their typical website session, the tasks they're interested in, and how they complete them. All along, we looked for ways to improve each interaction and improve the flow. The number of ‘small wins’ we made at this stage really added up.

We had been browser testing along the way, but at this stage we really submerged ourselves in it until every bug was worked out, from the iPad3 down to Internet Explorer 6. The app also makes heavy use of print stylesheets. Every page prints clearly. Even the lightboxes. Even in IE6.

Release Party

After fixing every bug and double-checking every browser, we could see the fruits of our labor taking shape. But we still had one large problem to address before the big unveil: announcing a website overhaul to a conservative, non-technical user base. gasp!

We did this in several waves. First we designed and sent an HTML email announcing the imminent changes (sent with Campaign Monitor). Next, we updated our in-site support docs with new screenshots and training on features. When we pushed the updated site live, we did it after hours (most of our users are in North America) with the whole web team present to put out fires. As one last measure of support, the development team (myself included) worked as customer support for a while.

A Scalable Interface Language

As we planned and prototyped new features, I consolidated the app's interface elements and organized them into HTML & CSS classes that can be mixed and matched. This sped up the design process, made interface elements more consistent, and created easy-to-use design patterns for our back-end developers.

The Website is Alive!

In the weeks after the launch, we received an overwhelmingly positive response from our users. Comments ranged from “The website is more useful now” to “the website is simply a pleasure to use”. But one of the most rewarding aspects of working on this app is that it's evolving, we work on it frequently throughout each year, and I'm helping guide its course.
